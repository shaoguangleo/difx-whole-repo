DiFXControl.MonitorThread Class Reference

The MonitorThread class provides a threaded receiver for the "packet protocol" of the DiFX server. Given a (connected) socket, it will consume complete packets as they are received and alert callback functions (set by methods setCallback() and setFailCallback()) when new data or failures are detected.

MonitorThread inherits the Thread class, so it needs to be started like a normal thread. The following code shows a typical use:

1 # Create an instance of MonitorThread with an existing socket.
2 monitorThread = MonitorThread( mySocket )
3 
4 # Set callbacks with defined functions/methods that are to be called when
5 # new packets are received or socket failures occur. See the method
6 # documentation for what the callback functions should look like, and
7 # the DiFXControl.Client class for examples. If you don't set the callback
8 # functions the thread will still run, but it won't do anything useful
9 # and will consume and throw away incoming data.
10 monitorThread.setCallback( myPacketCallback )
11 monitorThread.setFailCallback( myFailCallback )
12 
13 # Start the thread.
14 monitorThread.start()

MonitorThread was created to be used by the DiFXControl.Client class, but it may have other uses.

Non-Threaded MonitorThread

MonitorThread is also, oddly enough, useful in a non-threaded mode. If run as a thread it will be continually waiting on the select() line in the newPacket() method. This might be annoying for other threaded functionality, or if you wish to run your own select() call. An example of such a use is in the DiFXBusy application (when running with the Fltk GUI) - it has its own select() function (part of the Fltk package) that triggers a callback whereupon the newPacket() method of this class is called (including another select(), which returns immediately).

def DiFXControl.MonitorThread.newPacket (   self)

Read a single data packet by interpreting the the low-level packet protocol of the difxServer, which follows the pattern:

1 int packetID # 4-byte integer, network byte order
2 int packetSize # 4-byte integer, network byte order
3 string packetData # "packetSize" bytes in length

See the DiFXControl.Client method for a list of packetIDs.

Some effort is made to detect broken sockets and identify at what stage of the process they were broken - a broken socket will trigger a call to the "fail callback" if the user has set one (see setFailCallback()).

When (proper) packet data are received, a call is made to the "callback" function, if the user has set one (see setCallback()).

This is designed as an internal function, called repeatedly by the endless loop in the main thread run() method. However you can call it on your own if you wish to consume packet data.

def DiFXControl.MonitorThread.run (   self)

This is the main loop of the thread, triggered by a "start()" function call. This loop reads packets until something breaks (which might be caused by a user stop() call). Beyond looking for such trouble, this method simply calls the newPacket() method repeatedly.

Set the callback function for received packet data. The function should take two arguments, an integer "packet ID", and the data associated with it (a string). A sample use is below:

1  def myPacketCallback( self, packetId, data ):
2  print "received packet ID", packetId
3  print "its length in bytes is", len( data )
4 
5 ...
6 monitor.setCallback( myPacketCallback )

callbackFunctionThe function that will be called when a socket failure is detected.

Set the callback function for socket failures - the function should take no arguments. If you don't use this, the socket will simply stop on a failure.

def DiFXControl.MonitorThread.stop (   self)

This method should stop the main thread. It has a "sleep" that is twice the timeout of the select call in the "run" portion of the thread to give that endlessly-looping thread a chance to play out.
